Comprehensive HTML Feature Showcase

This document demonstrates every HTML element supported by the AIPDF HTML converter. It tests the full rendering pipeline from HTML input through semantic XML to production-quality PDF output, covering headings, paragraphs, lists, tables, code blocks, blockquotes, and figures.

Paragraphs and Text

The AIPDF format preserves document structure semantically. Paragraphs are the primary text block, capturing prose content. Long paragraphs are word-wrapped using precise Helvetica character-width metrics to ensure accurate line breaks in the rendered PDF output.

Multiple consecutive paragraphs are each emitted as individual semantic paragraph blocks. Each block receives a unique stable ID for cross-referencing and ordered extraction. The semantic layer stores no embeddings, model output, or executable content.

AIPDF files are valid PDF 1.7 documents. Legacy PDF readers display the visual content stream while ignoring the embedded associated file. AI-aware parsers detect the marker, extract the Brotli-compressed payload, and parse the structured XML directly.

Key features of the AIPDF format:

  • Brotli-compressed semantic XML embedded as an associated file
  • Full PDF 1.7 compliance — readable by any PDF viewer
  • Zero OCR required for machine-readable consumption
  • Schema-validated structure with stable block IDs
  • Three SDK surfaces: Rust (core + CLI), Python, and TypeScript
  • Export to XML, Markdown, Markdown AST JSON, and ONTO columnar format

Ordered Lists

Steps to produce an AIPDF document:

  1. Author content in XML, Markdown, HTML, or Typst format
  2. Run the CLI: aipdf build input.html --render full -o output.aipdf
  3. Distribute the .aipdf file — legacy readers see the PDF, AI readers extract the XML
  4. Use the CLI, Python SDK, or TypeScript SDK to query the semantic layer
  5. Export to your preferred format for downstream processing

Code Blocks

Rust Example

use aipdf::{build_aipdf, BuildOptions, RenderMode, PageOptions};

fn main() -> aipdf::Result<()> {
    let xml = std::fs::read_to_string("document.xml")?;
    let pdf = build_aipdf(&xml, &BuildOptions {
        title: "My Document".to_string(),
        render: RenderMode::Full,
        page: PageOptions::a4(),
        ..Default::default()
    })?;
    std::fs::write("document.aipdf", pdf)?;
    Ok(())
}

Subsection: Security Model

The AIPDF security model is based on three principles that prevent payload injection attacks:

  • Sanitization rejects DOCTYPE declarations, inline script content, and prompt injection markers
  • Structural validation enforces schema compliance before embedding
  • The semantic layer never contains embeddings, model output, or executable content

Disallowed Markers

The sanitizer rejects any XML payload containing the following strings regardless of case:

  • DOCTYPE
  • xml-stylesheet processing instruction
  • script elements
  • PDF JavaScript and Launch action types
  • LLM injection directives (prompt prefix strings)
  • AI system override strings (instruction injection patterns)
Implementation Detail

Character widths for word-wrap are measured in 1/1000 pt units using the complete Adobe Helvetica metrics table for all ASCII printable characters. Both regular and bold weights are supported with separate width tables.

PDF Object Structure

The minimal render mode produces exactly 9 PDF objects. The full render mode produces a dynamic number of objects: catalog, page tree, one page object per page, font resources, content streams, XMP metadata, embedded file spec, and the Brotli-compressed semantic stream.

ONTO Export Format

The ONTO format is a columnar, token-efficient representation designed for LLM ingestion. It uses pipe and caret delimiters for table data and applies scalar encoding rules to prevent delimiter collision in cell content.

Implementation Notes

The HTML converter uses the scraper crate (html5ever-based) for spec-compliant DOM parsing. This handles malformed HTML, implicit element closure, and Unicode normalization correctly — all critical for real-world document conversion where authors do not always produce well-formed markup.

Conclusion

The AIPDF format achieves its core design goal: a single file that serves both human readers via standard PDF rendering and AI systems via embedded structured XML. The semantic layer adds minimal overhead while enabling precise, OCR-free content extraction across all supported output formats.

The production-quality PDF renderer produces multi-page documents with styled headings, word-wrapped paragraphs, grid-based tables, syntax-highlighted code blocks, bulleted and numbered lists, citation sidebars, and footnote annotations — all using only base-14 PDF fonts with no external dependencies.
